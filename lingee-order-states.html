Lingee 订单状态机 · 开发落地依据

海外自助购买(Professional)。卡/FPX/钱包即时支付。本文档定义订单状态、流转条件、每个状态页展示什么、下一步引导什么。已与业务确认,直接作为开发依据。

适用范围:本状态机针对 ORDER(订单)。Professional 自助购买点 "Continue to payment" 即进入 pending_payment。Premium / 大客户走销售漏斗(LEAD 线索 → OPPORTUNITY 商机),只有赢单付款、生成 ORDER 后才进入本状态机;赢单前的跟进状态在销售域(见数据模型),不在此。

⇄状态流转图

stateDiagram-v2 [*] --> pending_payment: 点 Continue to payment
(建单 + 同意存证) pending_payment --> under_review: 支付成功(Webhook) pending_payment --> pending_payment: 支付失败(payment_failed 提示页)
/ 稍后再说(留单) pending_payment --> cancelled: 客户主动取消 pending_payment --> closed: 14天未付,自动关闭 under_review --> provisioning: 风控/合规通过 under_review --> rejected: 命中制裁/受限国家 provisioning --> completed: 履约完成(建租户/加席位/续期/减配) provisioning --> activation_failed: 履约失败 rejected --> refunded: 全额退款 activation_failed --> completed: 人工补做成功 activation_failed --> refunded: 全额退款 completed --> [*] cancelled --> [*] closed --> [*] refunded --> [*]
关键:payment_failed 不是独立订单终态,而是 pending_payment 下"某次支付尝试失败"的提示页,订单仍待支付、可重试(14天内)。cancelled(主动取消)与 closed(超时关闭)分开记录,便于区分"主动放弃"与"超时流失"。

≡状态定义

status含义进入条件类型
pending_payment待支付:订单已建未付点 Continue to payment进行中
under_review风控审核中:已付款待放行支付成功 + 进风控闸进行中
provisioning履约中:建租户 / 加席位 / 续期 / 减配风控通过进行中
completed已完成:该订单履约动作完成、服务可用履约成功终态(成功)
rejected已拒绝:合规不可售命中制裁/受限→ refunded
activation_failed履约失败:已付款未完成建租户/变更异常→ completed / refunded
cancelled已取消:客户主动放弃客户点取消并确认终态
closed已关闭:超时未付pending 超 14 天终态
refunded已退款(+ 环境回收)rejected/activation_failed/开通后退款 完成终态
payment_failed = 一次支付尝试的结果(展示用),非订单状态字段值。订单此时仍为 pending_payment。
为什么成功终态叫 completed 而不是 active:active 是订阅 / 租户的生命周期状态(SUBSCRIPTION.status / TENANT),表示"服务在线";订单的成功终态表示"这笔订单的履约动作做完了",对四种 order_type 都成立——new_business 建租户、expansion 加席位/点数、renewal 延期、contraction 减配,做完都置 completed(只有新购独有"建租户"这一步)。两者分层,避免把"订单完成"和"租户 active"混为一谈。
进入 refunded 时一并触发环境回收(KDM 编排,"停用→保留期→销毁"):合规拒绝时环境未建无需回收;履约失败清理半成品;开通后退款则 SUBSCRIPTION→canceled、TENANT 生产+沙箱 suspended→保留期(默认 30 天,只读供导出)→terminated(删/匿名化)、SEAT→revoked、增购点数 POINT_LEDGER 写 revoke 清零、退订 Stripe 订阅防再扣、发票红冲、写 ORDER_EVENT+AUDIT_LOG+退款确认通知。详见数据模型 REFUND 说明。

▤各状态:展示什么 + 引导什么

状态展示内容主操作 → 去向次操作
pending_payment页 订单号、配置摘要、税前金额、14天倒计时、已发恢复链接到邮箱 继续支付 → Stripe 收银台 稍后再说(离开,留单 pending)
取消订单(确认)→ cancelled
payment_failed页
(pending 下的提示)		 明确"未扣款"、失败原因、金额 重试支付 → 回收银台(换卡/FPX/钱包) 返回(订单仍 pending,14天内可再来)
under_review页 已收款确认、"正在确认订单,通常几分钟"、订单号。
措辞中性:不暴露"风控/制裁"字样		 无客户操作(等待)。"完成后邮件通知" —(多数订单秒过,客户可能看不到此页)
rejected页 "无法在你的地区/为该账户提供服务"(不解释具体制裁原因)、款项将全额退回、客服 联系客服 —
provisioning页 转圈 + "正在准备你的生产/沙箱环境,通常不到1分钟" 无(自动跳 completed) —
activation_failed页 款已收但开通遇到问题、团队已介入、会补开或全额退款、订单号、客服 联系客服 查看订单
completed页 ✓ 成功、生产/沙箱两个入口、发票&登录已发邮箱 进入生产环境 进沙箱 / 管理订阅
cancelled后台/邮件 订单详情显示"已取消" 重新下单 —
closed后台/邮件 再次打开旧链接时:"订单已过期,请重新下单" 重新下单 —
refunded后台/邮件 管理后台订单详情显示"已退款 + 金额 + 退回方式" — —
页 = 需要做成购买流程中的状态页;后台/邮件 = 在管理后台/邮件体现,不做独立购买页。

▣统一"订单信息块"(所有状态页复用)

每个状态页都用同一个订单信息块,保证一致、避免各写一套。基础 4 字段恒有,按状态增减附加行。
字段所有页取值/来源
订单编号 Order no.✓ 恒有ORDER.id(对外号 LG-OS-…)
下单时间 Created✓ 恒有ORDER.created_at · 马来时间 GMT+8 · YYYY-MM-DD HH:mm (GMT+8)
购买内容 Items✓ 恒有版本 + 席位 × 周期(+ 增购点数)
金额(不含税) Subtotal✓ 恒有ORDER.amount_subtotal · USD · pre-tax
↓ 按状态附加
有效期 Reserved until仅 pending_paymentexpires_at(GMT+8)+ 剩余天数
失败原因 Reason仅 payment_failedStripe 返回 + "未扣款"提示
已付 Paid(含税)已付款类:under_review / completed / activation_failed / rejectedORDER.amount_total · Stripe 回传含税实付
服务期 Term仅 completedstart → end(GMT+8 日期)
退款 Refundrejected / activation_failed → refunded退款状态/金额/原路
金额规则:未付款状态(pending/payment_failed)只显示"不含税小计";已付款后的状态额外显示 "Paid = Stripe 回传的含税实付"。前端永不自算税。
时区规则:所有时间存 UTC、对客户展示马来时间 GMT+8 并显式标注(法律/开票/对账口径统一以马来时间为准)。

⚙开发落地要点

  • • 建单时机:点 "Continue to payment" 即建 pending_payment 订单 + 落同意存证(谁/时间/协议版本/IP),再创建 Stripe Checkout Session。
  • • 有效期:pending_payment 起算 14 天,定时任务到期置 closed。
  • • 开通由 Webhook 驱动:checkout.session.completed → 进 under_review → 过风控闸 → provisioning → completed。Webhook 幂等。
  • • 已付款的异常都要给退款出口:rejected 与 activation_failed 必须明确告知"款项退回",避免"钱付了没东西"的投诉/拒付。
  • • under_review 措辞:对客户中性("Confirming your order"),不显示"制裁/风控"等内部术语。
  • • cancelled vs closed 分开存,用于区分主动取消与超时流失的运营分析。

0业务背景与目标

做什么:一个海外自助购买灵基(Lingee)的站点 —— 选产品 → 填信息 → 下单支付 → 自动开通环境。

核心诉求:① 客户体验顺畅、自助完成;② 法律/数据/税务/资金四类合规;③ 支付后自动开通产品;④ Stripe 处理收款及相关支付工作。

  • 两个版本:Professional(自助,$15/席/月 或 $180/席/年,起售 5 席)、Premium(面议,起售 100 席,多 AI 财务部 + CEO 办公室治理能力)
  • 点数:$300/万点($0.03/点),最小 1 万点;每席每月赠送 500 点(仅生产、当月有效)
  • 每套产品:1 生产租户 + 1 沙箱租户(隔离),公有云;可连客户私有 ERP(星瀚/星空)
  • 收款:Stripe(卡/FPX/GrabPay/Apple/Google Pay/Link),USD 定价 → MYR 结算
主数据源(SoT)原则:灵基商业化后台是产品/订单/订阅/席位/点数/权益的权威源;Stripe 是支付执行层(支付结果/税额/结算回传以 Stripe 为准)。两边以 stripe_*_id 映射,靠 Webhook + 定时对账保持一致。下单展示用灵基自己的产品目录,不实时查 Stripe。

1.2 Lingee 点数规则
  • 赠送:每席每月 500 点 → 仅生产环境、按月派发、当月有效、过期作废、不结转
  • 增购:$0.03/点($300/万点),最小 1 万点 → 生产 + 沙箱均可用、可分配到人(个人额度上限内共享)、即时发放、随主合同租期有效、过期作废
  • 记账:点数用流水账本(POINT_LEDGER),赠送/增购分别按 expires_at 失效;余额由账本汇总
1.3 购买旅程

A. Professional 自助(PLG,无销售):

选产品/配置 → 填信息+勾选协议 →〔点 Continue:建待支付订单 + 同意存证〕→ Stripe 嵌入收银台付款
→ 支付成功(Webhook)→ 风控/合规闸 → 开通中(建租户/权益/点数)→ 开通成功(进入产品)

B. Premium / 大客户面议(销售漏斗,quote-to-cash):

点 Contact Sales / 申请试用 → LEAD(线索):自动评分+建议、归并到已有客户
→ 销售跟进 人工 Qualify → OPPORTUNITY(商机):报价/合同/定制 Stripe 发票
→ 赢单付款 → 转成 ORDER(订单)(此刻才进 9 状态订单机)→ 开通 → SUBSCRIPTION/TENANT
  • 建单时机:点 "Continue to payment" 即创建订单(pending_payment)并落同意存证,再创建 Stripe Session
  • 支付:Stripe Embedded Checkout(站内嵌入,卡号不过我方服务器);账单地址、税、支付按钮均由 Stripe 收银台处理
  • 开通:由 Stripe Webhook 异步驱动(不在前端回调里开通),先过风控闸再调灵基 KDM 开通
对象主线(一条链,但多入口):LEAD 线索 →(人工鉴定)→ OPPORTUNITY 商机 →(赢单付款)→ ORDER 订单 →(开通)→ SUBSCRIPTION + TENANT 订阅/租户。全程都挂在 ACCOUNT 客户 + CONTACT 联系人 上,串成连续档案。
三种入口都合法(层级是"最长路径"不是"强制路径"):① 自助直接建订单——Professional PLG,无销售,不建线索/商机;② 销售直接建商机——大客户线下/转介绍谈来,意向已确认,OPPORTUNITY.lead_id 可空,跳过线索;③ 线索逐层转化——官网询价/申请试用/市场留资,先 LEAD → Qualify → 商机。规则:任一层都可作起点,但不能回退(订单不倒退回商机);无论从哪进,都先归并到同一 ACCOUNT。
1.4 订单状态 & 状态页规范

状态(ORDER.status,9 值):

状态含义下一步引导
pending_payment待支付(14天有效)继续支付 / 稍后 / 取消(→cancelled)
under_review已付款,确认中(风控)等待,邮件通知(措辞中性)
provisioning履约中(建租户/加席位/续期/减配)等待(转圈,自动跳完成)
completed已完成(履约动作完成)进入生产/沙箱
rejected合规拒绝,不可售联系客服,全额退款
activation_failed已付款但开通失败联系客服,补开或退款
cancelled客户主动取消重新下单
closed超时未付自动关闭重新下单
refunded已退款后台/邮件体现

payment_failed 不是 status 值,是 pending 下"一次支付尝试失败"的展示(记在 PAYMENT 实体)。

状态页统一信息块(我的设计主张):

  • A. 恒显示(订单身份):Order no. · Order date (GMT+8) · Items(版本·周期·席位·增购点数 合并一行)· Subtotal (pre-tax, USD)
  • B. 按状态追加:有效期(仅 pending)· 失败原因(payment_failed)· Paid 含税实付/Stripe(已付款类)· 服务期(completed)· 退款(rejected/activation_failed)
主张要点:① 配置细节(Billing/Seats/Gifted/Add-on)合并进 Items 一行,不平铺——状态页看"到哪了"而非重看配置;② 未付款只显示税前小计,已付款才显示 Stripe 回传含税 Paid(前端不算税);③ Gifted units 放成功页即可。

维度要点

⚠️ 制裁/出口管制:中国技术 + 马来主体 + 美元 + Stripe + AI 产品 → 大概率受美国 OFAC/出口管制约束,必须专业律师评估(不只是名单筛查)。受限国家在下拉排除 + 风控闸拦截。
后台覆盖:四维在 Lingee BC 后台均有运营视图——法律:Compliance&Risk(筛查)+ Consent Log(同意存证)+ Agreements&Versions(协议版本库,含子处理方清单作为 DPA 一部分);税务:Invoices·MyInvois;数据:租户驻留区域在「订阅与租户」详情展示、子处理方清单并入协议库 DPA(不单设"数据合规"页);资金:Reconciliation(对账)。政策制定/律师定稿/DPA 签署属平台/法务侧,后台只做运营可见与执行。
2系统设计
2.1 架构与边界(三层)
  • SoT 灵基 Lingee BC(商业中心 / 商务层):产品/SKU、订单、订阅、合规、Webhook、履约对账(流水↔订单匹配)、通知
  • 灵基 KDM(产品/履约层):建生产+沙箱租户、权益开关、点数派发与计量、ERP 连接器
  • Stripe 支付执行:Checkout、Billing、发票、税额计算、退款
  • 金蝶 ERP(财务大后端):收款记账、MFRS15 收入确认、财报、期末三方对账
  • 外部 LHDN MyInvois、制裁名单、协议/DPA 版本库

支付成功 ≠ 立即开通:Webhook → 履约对账(流水↔订单匹配)+ 风控/合规闸(Radar+制裁+受限国家) + TID/SN 对齐闸(问 KBC 判增购/新购,防开两个租户)→ 都通过才调 KDM 开通。履约对账在后台、秒级、不等 ERP;财务记账/对账在金蝶 ERP、开通后异步、不阻断。详见 架构与时序。

COMPLIANCE_CHECK灵基制裁/出口/税号校验明细
集成WEBHOOK_EVENTStripe幂等去重(event_id 唯一)

MESSAGE_LINK(消息关联·多态)灵基一条消息挂 N 个任意类型对象(account/order/invoice/subscription/support_request,可扩展)+ linked_by(system/human);多关联与 category 正交(“关于谁”vs“是什么”)

SUPPORT_REQUEST(售后工单)灵基客户就已购服务发起的问题(开通/退款/开票);以客户为中心、按权益(订阅)为单位,可下钻到订单;对话流派生自消息总账;与售前 LEAD 分开
合规审计AUDIT_LOG(操作审计)灵基人的敏感操作(退款/改开票/放行风控/改版本/导出);与状态历史分开
审批APPROVAL(审批记录)灵基统一审批模型:退款/风控放行/红冲/协议发版;双人复核(发起≠审批)+ 审批意见;支撑「待审批」工作台
关键建模决策:① 产品=PRODUCT、变体价格=SKU,分开;② 币种从 SKU 带出(USD),结算币种 MYR 来自 Stripe 账户/payout 回传;③ payment_failed 用 PAYMENT 实体表达,不污染订单 status;④ 合规/风控明细在 COMPLIANCE_CHECK,订单只存汇总 compliance_status;⑤ 金额用整数(分),点数用流水账本;⑥ 履约域(SUBSCRIPTION/TENANT/SEAT/POINT_LEDGER)SoT=灵基 KDM,BC 仅引用与展示、不存权威字段、不在 BC 改(故 BC 后台不设点数账本页)。完整字段见 数据模型。

"何时写日志 vs 落实体":有业务/法律/财务意义的状态 → 落实体字段(status/close_reason/时间戳);每次流转 → 写 ORDER_EVENT 审计日志(只增不改);纯技术细节(Webhook原始包/重试)→ 进日志系统。

2.3 税务与支付机制
  • 支付集成:Stripe Embedded Checkout(站内嵌入)。前端 Step2 点继续 → 后端建 Session(ui_mode:embedded)→ 前端挂载收银台
  • 税:Stripe Tax 不支持马来主体 → 用手动 Tax Rate + dynamic_tax_rates。已建 txr_1Tfc93AYtN4d3VHPgvmCZ3fN(Malaysia SST 8%);Stripe 按账单地址自动匹配:马来套 SST、海外零税率。前端不判断、不算税
  • 金额:未付款显示税前小计;含税总额由 Stripe 收银台显示;付款后 Webhook 回传 amount_subtotal/tax/total 入库
  • 币种:USD 定价收款 → Stripe 换汇 MYR 结算到马来账户(汇率+换汇费,财务关注)
  • 时区:存 UTC,展示马来时间 GMT+8 并标注;开票/对账以马来时间为准

详见 Stripe 对接 与 动态字段清单(禁止前端硬编码)。

2.4 开通与履约
  • Webhook checkout.session.completed → 验签/幂等 → 履约对账(流水↔订单匹配) → 风控闸 → TID/SN 对齐闸(问 KBC:该客户已有该产品订阅?→ 增购挂已有租户 / 新购建租户;冲突转人工、KBC 超时进待开通队列,防"线下新购+线上增购开出两个租户") → 都通过则调灵基 KDM
  • KDM:建 1 生产 + 1 沙箱租户(隔离)→ 按版本开权益 → 派发赠送点数 + 注入增购点数 + 启用计量
  • 赠送点数:按月定时任务派发 500/席,月底清零
  • 开通失败 → activation_failed(已付款,补开或退款,不让客户钱付了没东西)
  • 财税:生成发票 → 上报 MyInvois;开通后异步推计费/履约事件给金蝶 ERP 做收款记账 + MFRS15 收入确认 + 期末三方对账(不阻断开通)
  • 退款后环境回收(KDM 编排,"停用→保留期→销毁"):退款成功触发 → SUBSCRIPTION 置 canceled、TENANT 生产+沙箱 suspended → 保留期(默认 30 天,只读供导出/反悔)→ terminated(删除/匿名化数据)、SEAT 置 revoked、增购点数 POINT_LEDGER 写 revoke 清零、退订 Stripe 订阅防再扣、发票红冲(MyInvois credit note)、ERP 冲减收入,并写 ORDER_EVENT + AUDIT_LOG + 发退款确认通知。合规拒绝时环境未建、无需回收;开通失败清理半成品。

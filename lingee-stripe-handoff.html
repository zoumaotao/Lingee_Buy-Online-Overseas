币种与结算(钉死,勿各写各的):
presentment currency = USD——Stripe Price 对象按 USD 建,客户卡上被扣的是 USD(发卡行再换成持卡人本国币,与我方无关)。
结算到商户 = MYR,分两个数:① 毛换汇 = 含税总额 × Stripe 换汇汇率;② 净到账 payout = 毛换汇 − Stripe 手续费(真正打到马来银行账户的钱)。后台「结算到账(MYR)」列展示净额,详情页拆 毛/费/净三项。
退款原路退回原始 USD 扣款,不按退款当天新汇率重算(实时汇率只在下单付款那一刻生效)。
发票币种:MYR 为主(马来主体的 SST 以 MYR 向 LHDN 申报)+ USD 辅(客户计价币种)并列展示;按开票日汇率换算 MYR(LHDN 要求外币发票列示开票日 MYR 等值;汇率源用 BNM 开票日参考汇率)。
⚠️ 这个发票汇率只影响"报税口径",不影响资金:客户实付 USD(发卡行换汇)、Stripe 按 payout 当时汇率结算 MYR 给你——两者与发票汇率无关。所以发票上的 MYR(开票日汇率)与银行实收 MYR(payout 日汇率、已扣手续费)不一致是正常的,是两个口径。

4 研发:创建 Checkout Session(后端)

客户在 Step 2 点"Continue to payment"时调用。关键参数如下(Node 伪代码)。
// POST /api/checkout — 创建嵌入式 Checkout Session
const session = await stripe.checkout.sessions.create({
  ui_mode: 'embedded',            // ← 站内嵌入(关键)
  mode: 'subscription',          // 订阅;纯点数包用 'payment'
  line_items: [
    { price: 'price_1Tfbl4AYtN4d3VHPfEz9k9oK', quantity: seats,
      dynamic_tax_rates: ['txr_1Tfc93AYtN4d3VHPgvmCZ3fN'] },   // 年订阅 price;月付用 price_1TfbgGAYtN4d3VHPV61VuyL5。Stripe 按账单地址匹配:马来→SST,海外→零税
    { price: 'price_1Tfbl4AYtN4d3VHPTwBNxz27',  quantity: unitPacks,
      dynamic_tax_rates: ['txr_1Tfc93AYtN4d3VHPgvmCZ3fN'] }
  ],
  customer_email: buyerEmail,        // Step2 收集的邮箱
  customer_update: { address: 'auto' }, // 用账单地址匹配税率(dynamic tax rates 必需)
  tax_id_collection: { enabled: true }, // 收税号(零税率/反向征收依据)
  billing_address_collection: 'required',  // 账单地址在 Stripe 收银台收
  return_url: 'https://buy.lingee.app/done?session_id={CHECKOUT_SESSION_ID}',
  metadata: { orderId, edition: 'professional', seats, term } // 自家订单关联
});
return { clientSecret: session.client_secret };
✅ 用 dynamic_tax_rates(非 automatic_tax,Stripe Tax 不支持马来):Stripe 按客户在收银台填的账单地址自动匹配税率——马来地址套 SST 8%,海外无匹配=零税率。前端/Step2 不判断国家、不做一致性校验、无退款逻辑。席位数=quantity;metadata 带订单号供 Webhook 关联。

6 研发:Webhook 事件订阅清单(在 Stripe 端点勾选)

在 Stripe Dashboard → Developers → Webhooks 新建端点时,只勾下表这些事件(不要选 “all events”,噪音大且费配额)。每个事件到达后按“动作 / 落库实体”处理;全部先验签 + 按 event.id 幂等去重。
#Stripe 事件何时触发系统动作落库实体
结账 / 开通
1checkout.session.completed同步付款完成(卡 / 钱包)开通主入口:进 under_review → 履约对账 + 风控闸 + TID/SN 对齐 → 调 KDM 建租户ORDER · PAYMENT
2checkout.session.async_payment_succeeded异步付款成功(FPX 等延迟到账)同上——异步方式付款的开通入口(所以开通绝不能只靠前端 return_url)ORDER · PAYMENT
3checkout.session.async_payment_failed异步付款最终失败订单保持 pending_payment,提示客户重试;不开通PAYMENT
4checkout.session.expired结账会话超时未付释放预留,订单到期 closed(配合 14 天规则)ORDER
支付 / 风险
5payment_intent.succeeded扣款成功(权威资金事件)回填 amount_subtotal / tax / total + Radar risk_score/level 镜像成 COMPLIANCE_CHECKPAYMENT · COMPLIANCE_CHECK
6payment_intent.payment_failed一次支付尝试失败记一条失败 PAYMENT(含失败码),订单仍 pending_payment(不污染订单状态)PAYMENT
7charge.refunded退款已发生(回执)更新 REFUND→succeeded + 触发环境回收(KDM)+ 红冲发票 + 通知REFUND
8charge.dispute.created持卡人发起拒付建 DISPUTE,进举证流程;回流客户历史抬高同账户风险分DISPUTE
9charge.dispute.closed拒付结案(won / lost)更新 DISPUTE 结果;lost 收手续费,按结果决定是否回收环境DISPUTE
订阅 / 续费
10invoice.paid订阅续费扣款成功续期服务期、延展权益(续费在下一周期开始时交付给 KDM)ORDER(renewal)· INVOICE
11invoice.payment_failed续费扣款失败进 dunning(催缴)流程,订阅置 past_due;多次失败按策略停用SUBSCRIPTION
12customer.subscription.updated客户在门户改自动续费 / Stripe 改订阅镜像 auto-renew / 数量等状态(本系统只读镜像 Stripe)SUBSCRIPTION
13customer.subscription.deleted订阅到期 / 取消终止订阅置 canceled / expired,按策略走环境保留期SUBSCRIPTION
三条铁律:① 开通只由 Webhook 驱动,绝不依赖前端 return_url(异步付款会漏单);② 每个事件先验签(用端点 signing secret)再按 event.id 幂等去重(重复推送不重复开通 / 不重复退款);③ 资金真值以 Stripe 回传为准,前端/后台不重算金额、税、汇率。
事件分两类用途:触发动作类(1/2/7/8/10/11 会推进业务流程)与 镜像状态类(5/12/13 只更新本地只读镜像)。退款发起是后端主动调 POST /v1/refunds(见下一节),charge.refunded 只是退款发生后的回执。MyInvois 上报不在 Stripe 事件里——由开通/续费成功后另起流程经本地 PEPPOL 中间件完成。

7 研发:Webhook → 风控 → 开通(最关键)

// 监听的事件(完整清单见上一节;以下为核心处理流)
checkout.session.completed                  // 同步付款完成(card)→ 触发开通
checkout.session.async_payment_succeeded    // 异步付款成功(fpx)→ 触发开通
checkout.session.async_payment_failed       // 异步付款失败(fpx)
payment_intent.payment_failed               // 支付尝试失败 → 记 PAYMENT,订单仍 pending
charge.refunded                             // 退款 → 记 REFUND
charge.dispute.created                      // 拒付发起 → 建 DISPUTE,进举证流程
charge.dispute.closed                       // 拒付结案(won/lost)→ 更新 DISPUTE
invoice.paid                                // 续费成功
invoice.payment_failed                      // 续费失败 → dunning

// 处理逻辑
1. 验签(用 webhook signing secret) + 幂等(event.id 唯一约束去重)
2. 风控/合规闸:Radar 评分 + 制裁筛查 → 通过才继续,高风险转人工(拒付历史回流评分)
3. 调用灵基 KDM:建生产+沙箱租户、开权益、派发赠送点数 + 注入增购点数
4. 生成发票 → 经马来本地 e-invoice 中间件(PEPPOL)上报 LHDN MyInvois 取合规号(本地标准票 / 海外 self-billed);Stripe 不替你报 MyInvois,只出底票 PDF
5. 推计费/履约事件给金蝶 ERP 做 MFRS15 收入确认
6. 邮件:开通成功 + 登录链接
②③④⑤ 详见架构图与数据模型。开通前的风控闸是必须的,避免"钱收了产品也开了才发现欺诈/制裁"。
// 退款执行(后端调 Stripe Refund API,非后台手动点)
POST /v1/refunds  { payment_intent, amount }   // 默认原路退回原卡

// 谁触发 / 自动还是人工
合规拒绝 / 开通失败   → 系统自动全额退(我方责任,按协议必退,不等人工)
开通前客户取消       → 自动全额退(策略允许)
开通后退款           → 客服发起 + 人工审批(填审批意见),通过才调 API(审批意见写 AUDIT_LOG)

// 到账
受理后 REFUND.status=pending → 成功回传 charge.refunded → status=succeeded
退回原卡 5–10 个工作日(Stripe→发卡行时效,不可控,如实告知客户)

// 退款后:环境回收(KDM 编排,停用→保留期→销毁)
合规拒绝         → 环境未建,无需回收
开通失败         → 清理半成品租户,SUBSCRIPTION→canceled
开通后退款       → SUBSCRIPTION→canceled;TENANT(生产+沙箱)suspended
                  → grace_period(默认30天,只读供导出)→ terminated(删/匿名化)
                  SEAT→revoked;POINT_LEDGER 写 revoke 清零(增购点数失效)
                  退订 stripe_sub_id 防再扣;发票红冲 = 调 Stripe 开 credit note + 上报 MyInvois 取红冲凭证
                  写 ORDER_EVENT + AUDIT_LOG + 发 refund_confirmation 通知
退款金额、币种、原路均以 Stripe 回传为准;前端/后台不重算。charge.refunded 是退款发生后的回执,用来更新 REFUND 状态并触发环境回收,不是发起退款的方式。

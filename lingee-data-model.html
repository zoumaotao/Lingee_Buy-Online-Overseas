围绕"下单→收款→自动开通→合规"。每个实体标注主数据源(SoT):谁是权威。PK主键 FK外键 *必填,枚举值已穷举。

主数据源原则:灵基 灵基商业化后台是产品/订单/订阅/席位/点数/权益的权威源;Stripe 是支付执行层(支付结果/税额/结算回传以它为准);外部 如 LHDN。两边以 stripe_*_id 映射,Webhook + 定时对账保持一致。下单展示用灵基自己的目录,不实时查 Stripe。

3 · 销售域(线索 → 商机 → 订单)新增

自助购买(Professional)不经过本域,直接建 ACCOUNT + ORDER。Premium / 大客户面议从这里走:Contact Sales 提交 → LEAD → 鉴定合格 → OPPORTUNITY → 赢单付款 → 才转成 ORDER。
三入口(层级=最长路径,非强制):① 自助 → 直接 ORDER;② 销售直接建 OPPORTUNITY(lead_id 可空,大客户线下谈来);③ 线索逐层 LEAD→OPPORTUNITY→ORDER。任一层可作起点,不可回退,均先归并到同一 ACCOUNT。
LEAD · 销售线索(未鉴定的兴趣) SoT 灵基(或对接 CRM)
承接 Premium「Contact Sales」表单与市场留资。一条线索可能成交也可能是无效咨询,先由销售跟进鉴定。此时还没有 ORDER。

分层(与业内对齐):ORDER(交易意图,一个)→ PAYMENT(支付流水,每条 = 一次支付尝试,1 对多,= Stripe PaymentIntent:收这笔钱的全过程,内部可含多次扣款 Charge,并处理 3DS/SCA)→ REFUND / DISPUTE(挂在对应支付下,各自独立记录)。流水按"每次支付动作"走,不按订单走:这样失败重试、每次原因、退款/拒付都不丢。
流水不可改(关键,回应开发疑问):PAYMENT 是只增不改的账本。退款不修改原支付记录,而是新增一条 REFUND 流水;原 PAYMENT 仍 status=succeeded,只在其 refund_marker 上标 refunded/partially_refunded。所以"退款成功"绝不会把原流水改成"refunded 状态"——PAYMENT.status 永远只有 processing/succeeded/failed/canceled,退款是另一条记录。

退款怎么执行(谁触发 / 自动还是人工 / 调什么 / 几天到账):
三个发起源:① 系统自动——合规拒绝/开通失败时系统自建 REFUND(initiated_by=system);② 后台运营发起——客服在「退款与拒付」页或订单详情点"发起退款";③ 客户申请——客户在购买页 Contact support 提工单(SUPPORT_REQUEST,topic=refund)→ 客服核实后转成 REFUND。客户不能直接退款,只能申请;Stripe API 调用一律由后端发起。
退款由后端调 Stripe Refund API(POST /v1/refunds,传 payment_intent + amount,默认原路退回原卡)——不是在 Stripe 后台手动点。
① 合规拒绝(compliance_rejected)/ 开通失败(activation_failed)→ 系统自动全额退,无需人工:这是我方责任(没卖成 / 没交付),按协议必退,不让客户等人工。
② 开通前客户取消(pre_activation)→ 自动全额退(策略允许)。
③ 开通后退款 → 必须人工审批(客服发起 → 审批通过才调 API,记 initiated_by/approved_by 并写 AUDIT_LOG)。
到账:Stripe 受理后 status=pending,退款成功回传 charge.refunded → 更新本记录 status=succeeded;退回原卡通常 5–10 个工作日(Stripe→发卡行时效,我方不可控,如实告知客户)。
谁通知客户:钱由 Stripe 退;给客户的退款确认邮件由 Lingee 通知服务发(NOTIFICATION refund_confirmation,含订单/环境回收/数据删除说明)——只有我方有业务上下文。Stripe 自带的"退款收据"是纯支付层,建议关闭或仅作补充,避免与业务通知重复。
退款后:环境回收 + 数据联动(KDM 编排,SaaS 标准"停用→保留期→销毁",不立即物理删):
退款成功(charge.refunded)触发回收,按场景分:
• 合规拒绝(rejected):环境从未建(风控在开通前)→ 无需回收;仅 ORDER→refunded、写 ORDER_EVENT/通知。
• 开通失败(activation_failed):可能有半成品租户 → KDM 直接清理半成品,SUBSCRIPTION→canceled。
• 开通后退款:走完整回收 → ① SUBSCRIPTION.status=canceled;② TENANT 生产+沙箱 ready→suspended(立即停登录)→ 进 保留期 grace_period(默认 30 天,只读供导出/反悔) → 到期 terminated(删除/匿名化数据,记 terminated_at);③ SEAT.status=revoked;④ POINT_LEDGER 写一条 revoke 把增购点数余额冲减至 0(赠送点数本就当月失效);⑤ 退订 stripe_sub_id 防再扣费;⑥ 作废/红冲对应 INVOICE(MyInvois 开 credit note);⑦ 写 ORDER_EVENT(refunded)+ AUDIT_LOG(人工退款/回收操作)+ 发 refund_confirmation 通知(含"数据将于 N 天后删除")。
SoT:环境/点数回收在 KDM 执行;ORDER/退款状态在 Lingee BC;发票红冲与 ERP 冲减收入在财税侧。BC 后台只读展示回收进度。

6 · 订阅履约域 灵基 KDM

SoT = 灵基 KDM:本域(SUBSCRIPTION / TENANT / SEAT / POINT_LEDGER)的发放、扣减、计量、过期均在 KDM 执行与存储。Lingee BC 仅引用与展示(如订单详情看交付了什么),不存权威字段、不在 BC 改;点数明细查询去 KDM,故 BC 后台不单设点数账本页。
三个概念别混:SUBSCRIPTION 订阅=商务契约(买了什么版本/几席/到期);权益 entitlement="能用什么"的规则(版本→功能、治理模块开关、@智能体权限、点数额度),由 KDM 权益服务管;TENANT 租户=权益落地的容器(生产/沙箱环境)。订阅≠权益≠租户:订阅定义应得权益,权益是规则,租户是场所。

⚠️ 开通前 TID/SN 对齐闸(防"开出两个租户"):客户可能线下(KBC/SLG)新购又线上(LBC/PLG)增购,或反向。若开通前不对齐,会把增购误判为新购、给同一客户开出两个租户、权益分裂。规则:任何一方在调 KDM 建租户前,必须先向 KBC(客户+合同财务权威)对齐 TID/SN。
两层判定:① 入口初判——产品内·试用版购买→倾向新购;产品内·正式版购买→倾向增购(已知挂哪个 SN);官网/无上下文→待定。② 开通前终判(权威,以此为准)——用匹配链(已绑 TID → 马来 registration_no/tax_id;海外 税号→域名 → 模糊转人工)问 KBC + 查 KDM 是否已有该客户租户:已有 TID+对应产品 SN → 强制增购(挂已有租户,不另开);完全没有 → 新购(建 TID+SN+租户);初判与终判冲突 → 转人工复核。同客户同产品想开第二套独立订阅 → 必须人工复核确认,不自助直接开。
失败不死等(业内做法):对齐秒过→正常开通;KBC 超时/无响应→订单转待开通队列(客户见"处理中"、不报错),后台自动重试/人工兜底;冲突→人工复核。对齐链路走「集成事件 → 金蝶 ERP(KBC)」,失败可重试。开通(快、LBC 触发)与记账(慢、异步推 KBC)分离,记账不阻断开通。

与 Stripe 的协同:Lingee 不主动调用 Radar——Stripe 在收款时自动跑反欺诈评分,结果随 payment_intent webhook 带回,Lingee 镜像成一条 risk_score 检查记录(只读)。sanctions(看人)由 Stripe OFAC + 独立名单筛查服务做姓名比对;export_control(看货)是出口管制分类 + 目的地/用途判定,与 SDN 名单是两回事——二者是两条独立检查,不可合并。所有检查在开通前的风控闸汇总:全 pass 才放行开通;任一 manual_review → 订单停在 under_review,运营在「合规与风控」页人工 Clear/Block(走 APPROVAL 双人复核)。订单上的 compliance_status 是这些检查的汇总(任一 hit→blocked、有 manual_review→pending、全 pass→pass),明细在本表。
受限清单(法务维护,为权威来源;后台「合规 → 受限清单」基础资料页展示。字段:类型 / 对象 / 处理方式 / 说明 / 来源):
本清单管两类受限对象——国家级 与 个人/实体级,二者维护方式截然不同:
A. 国家级规则(几十个、法务给清单、本页人工维护):
① 红线·禁售(古巴/伊朗/朝鲜/叙利亚/俄罗斯/克里米亚/顿涅茨克/卢甘斯克):两端自动拦,无人工——购买页国家下拉不列出 + Stripe 按 IP/账单地址/发卡国三维拒付 + Radar 开 OFAC。不生成订单。
② 灰名单·禁售(白俄罗斯/缅甸/委内瑞拉/苏丹/南苏丹/也门/利比亚/索马里/中非/刚果金/马里/黎巴嫩/阿富汗/伊拉克):现阶段保守默认=同红线自动拦(宁可错杀,不背实时 SDN 筛查风险)。若要放行某正经客户,由法务把该国显式调到复核层。
③ FATF·加强监控(21 国,如越南/菲律宾/土耳其/南非等):不拦截。注意:Stripe 不认识 FATF 名单、不会自动加分——需我方在 Stripe Radar 配一条规则(如 card_country 命中这些简码则转 Review)才会"抬分/转人工";不配规则则无任何效果。本表的 FATF 行只是配 Radar 规则的源清单 + 法务监控建议的审计留痕,执行在 Stripe Radar。
B. 个人/实体级(OFAC SDN 名单,几千条、天天更新):不可能手工维护,本页不存全量名单——逐单由 Stripe OFAC + 筛查服务实时查,命中姓名即拦截该客户,与国家无关。本页可选保留少量本地黑名单(运营因欺诈/拒付手动封禁的几个账户)。
前后台怎么协同:① 国家规则在后台本页人工维护 → 喂给(a)前台购买页国家下拉(红/灰名单国家隐藏、客户根本选不到)和(b)Stripe 配置(IP/账单/发卡国拦截 + OFAC);② FATF 不自动拦、不自动加分——需把这些简码复制进 Stripe Radar 规则才生效(本表是源清单,改这里→同步 Radar);③ 个人/实体筛查是实时 API 调用、不是存表——前台永远看不到名单,只看到一个"允许下单的国家下拉";后台只存"国家规则 + 本地黑名单"。
存储粒度(为什么这样设计):国家表一国一行(以国家代码为主键,可逐条增删改 + 审计;重叠按取最严处理——同时在灰名单与 FATF 的国家按禁售,如南苏丹/刚果金/马里/也门/叙利亚)。本地黑名单一账户一行(我方自行封禁、手工维护)。唯独 OFAC SDN 不做一人一行——它几千条天天变,只能实时筛查、不存可编辑副本。
SDN 人工复核标准(仅当某单已被标 manual_review 时):核对客户(公司 + 具名联系人)是否真命中 OFAC SDN / 制裁名单条目——确认命中→Block;误报(重名/不同主体)→Clear。双人复核,结论写 decided_by/decided_at + AUDIT_LOG(risk_cleared/risk_blocked)。清单本身的增删改写 restricted_list_updated 审计。自动筛查不替代法务定稿(TODO L6)。

用法约定:① 界面元素只携带 key(如 data-status="activation_failed"),渲染时按语言查本表取文案;原型里用 data-en/data-zh 双语属性模拟同一机制。② key 一律 snake_case 英文,不随语言变;下游系统、Stripe metadata、KDM/ERP 接口都按 key 对接。③ 改文案或加值只在本表改,界面自动跟随。④ 与各实体字段表的枚举值保持一致——本表是汇总权威,字段表是局部引用。⑤ 枚举按域(实体.字段)分组,不要全局扁平化:同一 key 在不同域可有不同文案——例如 won/lost 在 dispute.status 是“胜诉/败诉”、在 opportunity.stage 是“赢单/输单”;pending 在合规、退款、发票各有侧重。渲染时按所在域取值(如 i18n['dispute.status']['won']),不要用全局扁平表;若工程上用了全局表,在冲突处就地覆盖文案。⑥ 纯技术标识不翻译:Stripe 事件类型(checkout.session.completed)、KDM/ERP 动作名(create_tenant/post_invoice)、支付失败码等按行业惯例保留英文 snake_case。
